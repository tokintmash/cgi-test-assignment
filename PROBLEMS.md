# Problems & Solutions

Issues encountered during development and how they were resolved.

| # | Problem | Cause | Solution | Date |
|---|---|---|---|---|
| 1 | N+1 query problem in RecommendationService — `findOverlapping` called twice per table (once for status, once for availability), causing ~41 queries per search | Each table triggered its own DB query inside a loop | Fetch all reservations for the date once with `findByDate`, then filter in-memory using Java Streams for overlap checks | 2026-03-08 |
| 2 | Users could book reservations in the past — no date validation on ReservationRequest | `date` field had `@NotNull` but no temporal constraint | Added `@FutureOrPresent` annotation to the `date` field | 2026-03-08 |
| 3 | Race condition in ReservationService — concurrent requests could both pass the overlap check and create double bookings | No transaction isolation between the overlap check and the save | Added `@Transactional(isolation = Isolation.SERIALIZABLE)` to `createReservation` | 2026-03-08 |
| 4 | Frontend lint failed in `BookingDialog` with `react-hooks/set-state-in-effect` | Local dialog form state (`guestName`, `duration`) was reset inside `useEffect`, which violates the configured React Hooks lint rule | Removed the effect-based resets and switched to remounting the dialog per booking session (`key` in `App.tsx`) so state resets cleanly without synchronous setState in effects | 2026-03-09 |
| 5 | Floor plan zone boundaries no longer matched the seeded table positions after earlier spacing tweaks | SVG zone frames were hard-coded while the table coordinates came from backend seed data, so `M7` and the lowest window table no longer fit the intended zones | Rebalanced the seeded table sizes and coordinates, converted `W5` into terrace table `T4`, and expanded the terrace zone to full width so every table fits its zone again | 2026-03-10 |
| 6 | Irrelevant tables recommended — e.g. W1 appeared for "Accessible + Near play area", non-private tables shown when searching Private, non-window tables shown for Window preference | Recommendation engine used soft scoring (partial preference match got a lower score but still appeared), and zone selection only adjusted score weight instead of filtering | Changed preferences and zone from soft scoring to strict filtering: tables must have **all** requested preferences and match the selected zone. Removed "Private area" from preference tags (redundant with zone select) and "Window" from zone dropdown (redundant with preference tag) | 2026-03-10 |
| 7 | Floor plan and recommendations pane height shrank when fewer results were returned, and all recommendations were visible without scrolling when many results returned | SVG sized itself naturally from its viewBox aspect ratio, making the grid row excessively tall. Multiple CSS approaches failed: `minmax(520px, auto)` didn't constrain the max, `aspect-ratio` didn't work in flex context, padding-top hack made the container too tall | Set `grid-template-rows: 740px` to lock both columns to a fixed height. Added `flex: 1; min-height: 0` on svg-wrapper and `height: 100%` on floor-svg so the SVG scales within the constrained space. Right pane scrolls via existing `overflow-y: auto` on `.recommendation-scroll` | 2026-03-11 |
| 8 | Terrace table combinations still appear in extreme cold — single tables are weather-filtered but combinations path skips the check | `findCombinations` has no `calculateWeatherPenalty > -1.0` filter, unlike the single-table pipeline at line 63 | Added weather penalty filter to the `findCombinations` available-tables stream | 2026-03-12 |
| 9 | `SearchRequest.duration` accepts negative values — a negative duration makes `endTime` earlier than `startTime`, causing all tables to appear available | No `@Min` validation on the `duration` field; compact constructor only defaults 0 to 120 | Added `@Min(30)` to `SearchRequest.duration` | 2026-03-12 |
| 10 | `preferenceMatch` score is always 1.0 for single-table recommendations — scoring dimension is dead code | The `hasAllPreferences` hard filter (line 61) already guarantees all preferences match before `calculatePreferenceMatch` runs | Dismissed — conscious decision. Hard filter was chosen over soft scoring (Problem #6). The constant 1.0 is harmless and `preferenceMatch` still varies for combinations | 2026-03-12 |
| 11 | Window zone silently excluded from search form zone dropdown | `App.tsx:128` filters out `'Window'` from the zones list (was made redundant with Window preference tag in Problem #6), so users cannot search by Window zone | Dismissed — intentional per Problem #6. The filter is the active mechanism; no unused remains found | 2026-03-12 |
| 12 | `WeatherServiceTest.returnsCachedOnSubsequentFailure` comment contradicts assertion — says "should return stale cached data" but `clearCache()` nulls the cache, and test asserts `null` | `clearCache()` wipes both `cachedWeather` and `cacheExpiry`, so the fallback path returns `null` not stale data; test name and comment are misleading | Renamed test to `returnsNullWhenCacheClearedAndApiFails`, fixed comment to match actual behavior | 2026-03-12 |
| 13 | Commented-out code and hardcoded fallback text in App.tsx weather banner | Line 407 has a commented-out weather string; line 408 shows `"Weather is nice."` as default text for normal weather with no penalty | Dismissed — default text is intentional; commented-out line is harmless | 2026-03-12 |
| 14 | No integration test for `WeatherController` (`GET /api/weather`) | All other controllers have MockMvc tests but `WeatherController` was added in Phase 4.5 without one | Added `WeatherControllerTest` with 200/204 handling (weather API may be unavailable in CI) | 2026-03-12 |
| 15 | No unit tests for table combination scoring or weather penalty on combinations | `RecommendationServiceTest` covers single-table and weather scenarios but has no tests for the `findCombinations` path | Added 5 combination tests: adjacent pair found, non-adjacent excluded, cross-zone excluded, terrace pair excluded in cold weather, warm-weather penalty is 0.0 | 2026-03-12 |
